# Sentinel v0.5 — Day-One DX Cycle (Design)

## Goal

Close the three biggest day-one DX gaps that block adoption against sqlx and
SeaORM: ergonomic pool arguments at the macro layer, macro-level streaming,
and a fixture-isolated test harness. Each gap is independently visible in a
new user's first hour with Sentinel.

## Non-goals (v0.5)

- Multi-database support (PG-only stays).
- Down/rollback migrations (forward-only by design).
- DB → struct scaffolder (`sntl generate entity`) — defer to v0.6.
- ActiveModel / mutation-by-dirty-flag.
- MockDatabase (no live PG) — defer.
- Cursor pagination helper at builder layer — defer.
- LISTEN/NOTIFY typed `query_subscribe!` — defer.
- Nullability inference v2 (CTE, subquery, UNION, window, lateral).

## Why these three

Ranked against the v0.5 entry criterion "every minute of friction kills
evaluators":

| Gap | Friction | Rival comparison |
|---|---|---|
| Macros require `&mut Connection` | Every call site has `pool.acquire().await?` boilerplate | sqlx + SeaORM both accept pool directly |
| No streaming at macro level | Forces drop to `CursorQuery` driver API | sqlx `.fetch()`, SeaORM `.stream()` |
| No `#[sntl::test]` harness | Every project hand-rolls `require_pg!` + `TRUNCATE` | sqlx `#[sqlx::test]` is one line |

After v0.5, Sentinel crosses ~85 % of sqlx surface and ~75 % of SeaORM
surface with a clearly better core. The remaining gaps are positional
(brownfield scaffolders, multi-DB) and can ride a slower cadence.

## PR-1 — Pool-as-first-class-arg

### Driver-side: `GenericClient` extension

```rust
// sentinel-driver — generic_client.rs (extended)
#[async_trait]
pub trait GenericClient: Send {
    async fn query(&mut self, sql: &str, params: &[&(dyn ToSql + Sync)]) -> Result<Vec<Row>>;
    async fn execute(&mut self, sql: &str, params: &[&(dyn ToSql + Sync)]) -> Result<u64>;
    async fn query_typed(&mut self, sql: &str, params: &[(&(dyn ToSql + Sync), Oid)]) -> Result<Vec<Row>>;
    async fn execute_typed(&mut self, sql: &str, params: &[(&(dyn ToSql + Sync), Oid)]) -> Result<u64>;
}

pub trait AsPool {
    async fn with_conn<R, F>(&self, f: F) -> Result<R>
    where
        F: for<'a> AsyncFnOnce(&'a mut Connection) -> Result<R> + Send,
        R: Send;
}

impl AsPool for Pool {
    async fn with_conn<R, F>(&self, f: F) -> Result<R>
    where
        F: for<'a> AsyncFnOnce(&'a mut Connection) -> Result<R> + Send,
        R: Send,
    {
        let mut conn = self.acquire().await?;
        f(&mut *conn).await
    }
}

// Blanket impl: any &P where P: AsPool implements GenericClient by
// auto-acquiring per method call.
#[async_trait]
impl<P: AsPool + Sync> GenericClient for &P {
    async fn query(&mut self, sql: &str, params: &[&(dyn ToSql + Sync)]) -> Result<Vec<Row>> {
        AsPool::with_conn(*self, |c| c.query(sql, params)).await
    }
    // ... other methods follow the same pattern ...
}
```

### Sntl-side: macro return-builder bounds

The query macros today emit code that calls `.fetch_one(&mut conn)` /
`.fetch_all(&mut conn)` / `.execute(&mut conn)` on a generated builder
struct. The builder methods currently require `&mut Connection`; widen
them to require `GenericClient`:

```rust
// Before:
impl QueryResult {
    pub async fn fetch_one(self, conn: &mut Connection) -> Result<Self::Row>;
}

// After:
impl QueryResult {
    pub async fn fetch_one<C: GenericClient>(self, mut client: C) -> Result<Self::Row>;
}
```

User-visible result:

```rust
// All three of these work identically in v0.5:
sntl::query!("...").fetch_one(&pool).await?;
sntl::query!("...").fetch_one(&mut conn).await?;
sntl::query!("...").fetch_one(&mut pooled_conn).await?;
```

### Tradeoffs

- **One acquire per macro call** on `&Pool`. Sequential queries against a
  pool will each check out independently. Users running transactions or
  multi-step flows should keep an explicit `let mut conn = ...`. Documented.
- **`AsyncFnOnce`** requires Rust 1.85+ — workspace is already at edition
  2024.
- **Cross-repo coordination:** sentinel-driver v2.2.0 must publish before
  sentinel v0.5.0 release. PR-1 is two coordinated PRs.

## PR-2 — `.fetch_stream()` at the macro layer

### Driver-side: `RowStream`

```rust
// sentinel-driver — connection/stream_impl.rs (new public surface)
pub struct RowStream<'a> {
    conn: &'a mut Connection,
    portal_name: String,
    columns: Arc<[ColumnDescriptor]>,
    batch_size: i32,
    buffer: std::collections::VecDeque<Row>,
    exhausted: bool,
}

impl<'a> Stream for RowStream<'a> {
    type Item = Result<Row>;

    fn poll_next(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Option<Result<Row>>>;
    // Implementation: pop from buffer; if empty and not exhausted,
    // send Execute(portal, batch_size). If server returns < batch_size
    // rows, mark exhausted and close the portal on next poll.
}

impl Drop for RowStream<'_> {
    fn drop(&mut self) {
        // Send Close(portal) + Sync; ignore errors.
        // Tx commits on stream drop (auto-tx opened at query_stream).
    }
}

impl Connection {
    pub async fn query_stream(
        &mut self,
        sql: &str,
        params: &[&(dyn ToSql + Sync)],
        batch_size: i32,
    ) -> Result<RowStream<'_>>;
}
```

PG portals are tx-scoped, so `query_stream` opens an implicit transaction
if none is active. Stream drop emits `COMMIT` (or `ROLLBACK` if an error
occurred during streaming).

### Sntl-side: macro extension

The codegen for `query!()` and `query_as!()` adds two methods to the
returned builder:

```rust
sntl::query!("SELECT id, name FROM users WHERE active = $1", true)
    .fetch_stream(&pool)              // default 50 rows / batch
    .await?;

sntl::query!("...")
    .fetch_stream_with(&pool, 200)    // custom batch size
    .await?;
```

Each variant returns `impl Stream<Item = Result<RowOfThisQuery>>` where
`RowOfThisQuery` is the same generated row struct the existing
`fetch_one`/`fetch_all` use.

`fetch_stream(&pool)` on an `&Pool` internally acquires once and holds
the `PooledConnection` for the lifetime of the stream, releasing on drop.

### Batch size default

**50 rows per Execute round-trip.** Configurable via `fetch_stream_with`.
Sweet spot for typical row sizes (~500 B each → ~25 KB per round-trip,
fits in one TCP segment).

### Tradeoffs

- **Stream borrows the connection.** Caller can't run other queries on
  the same connection until stream drops. Fundamental — Portals are
  tx-scoped.
- **Backpressure is implicit.** Caller's `.next().await` rate gates
  server-side row delivery. No buffering beyond the batch.
- **No owned-stream variant** (would need clone-able conn). Defer.

## PR-3 — `#[sntl::test]` fixture harness

### User-facing API

```rust
// Simplest form — fresh DB, no migrations, no fixtures
#[sntl::test]
async fn empty_db(pool: sntl::Pool) -> Result<()> {
    let n: i64 = sntl::query_scalar!("SELECT COUNT(*) FROM pg_database")
        .fetch_one(&pool).await?;
    assert!(n > 0);
    Ok(())
}

// With migrations
#[sntl::test(migrations = "./migrations")]
async fn with_schema(pool: sntl::Pool) -> Result<()> {
    sntl::query!("INSERT INTO users (id, name) VALUES (1, 'alice')")
        .execute(&pool).await?;
    Ok(())
}

// With migrations + named fixtures (loaded in declared order)
#[sntl::test(migrations = "./migrations", fixtures("users", "posts"))]
async fn full_setup(pool: sntl::Pool) -> Result<()> {
    let posts = sntl::query_as!(Post, "SELECT * FROM posts WHERE user_id = 1")
        .fetch_all(&pool).await?;
    assert_eq!(posts.len(), 3);
    Ok(())
}
```

Fixtures: `tests/fixtures/<name>.sql`. The macro discovers them relative
to the test's `CARGO_MANIFEST_DIR`, applies them in declared order inside
a single transaction after migrations finish.

### Implementation architecture

```
┌─────────────────────────────────────────────────────────────┐
│ #[sntl::test] macro expansion (sntl-macros)                 │
│  fn user_test() {                                           │
│    sntl::testing::run(                                      │
│      "user_test",                                           │
│      MigrationSource::Dir("./migrations"),                  │
│      &["users", "posts"],                                   │
│      |pool| async move { /* user body */ }                  │
│    )                                                        │
│  }                                                          │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│ sntl::testing::run                                          │
│  1. Resolve admin URL from SNTL_TEST_DATABASE_URL           │
│     (default postgres://localhost/postgres). If unset,      │
│     gracefully skip via eprintln! and return.               │
│  2. template_key = sha256(canonical migrations dir)[:8]     │
│  3. Once per process (OnceCell<TemplateDb>):                │
│     - DROP DATABASE IF EXISTS _sntl_tmpl_<key>              │
│     - CREATE DATABASE _sntl_tmpl_<key>                      │
│     - Connect, apply migrations end-to-end                  │
│     - ALTER DATABASE _sntl_tmpl_<key> WITH ALLOW_CONNECTIONS│
│       FALSE  (so it can be a TEMPLATE source)               │
│  4. Per test:                                               │
│     - test_db = _sntl_<sanitised_test_name>_<rand_4>        │
│     - CREATE DATABASE <test_db> TEMPLATE _sntl_tmpl_<key>   │
│     - Connect, apply fixtures in one tx                     │
│     - Build Pool with max_connections = 4                   │
│  5. Run user closure with that Pool                         │
│  6. On Ok: DROP DATABASE <test_db>                          │
│  7. On panic / Err: leak (no drop) and print test_db path   │
│     so debug shells can connect                             │
└─────────────────────────────────────────────────────────────┘
```

### Why template DB

- **~10 ms per test** via PG-level clone (vs seconds for re-running
  migrations).
- **True isolation:** each test gets its own DB; parallel-safe across
  tokio worker threads.
- **Hash-keyed:** editing migrations invalidates the template hash and
  rebuilds the template on next test run.

### Env vars

```sh
SNTL_TEST_DATABASE_URL=postgres://sentinel:sentinel_test@localhost:5432/postgres
SNTL_TEST_KEEP_DBS=1   # if set, never drop test DBs (debug)
```

If `SNTL_TEST_DATABASE_URL` is unset: skip with `eprintln!`. Same
contract as the existing `require_pg!` macro for backwards compatibility.

### File layout for users

```
your-project/
├── migrations/                 # Same folder sntl-migrate uses
│   ├── 20260514_120000_init/up.sql
│   └── …
├── tests/
│   ├── fixtures/
│   │   ├── users.sql
│   │   └── posts.sql
│   └── user_test.rs            # #[sntl::test] fns here
└── Cargo.toml
```

### Tradeoffs

- **Requires admin privileges on test PG** (CREATE/DROP DATABASE). CI
  fine; shared dev DB isn't — documented.
- **Test names must be filesystem-style unique** so DB names don't
  collide. Macro errors on duplicate names within a crate.
- **Slow first run per process** — template build takes "migrations
  time". After that, all tests share the template — per-test cost is
  ~10 ms clone + fixture-apply.
- **CI needs ≥ 1 PG instance** — recipe in `docs/testing-guide.md`.

## Testing strategy (across the three PRs)

**PR-1 — Pool-as-arg**
- `sntl/tests/pool_arg_test.rs` (live-PG): exercise `query!()` against
  `&Pool`, `&mut Connection`, `&mut PooledConnection` — all three return
  identical rows.
- Existing `sntl-cli/tests/*_test.rs` (which use `&mut conn`) MUST still
  pass — verifies backwards compatibility.

**PR-2 — fetch_stream**
- `sntl/tests/stream_test.rs` (live-PG): insert 1000 rows, stream them
  back, assert count + ordering matches.
- Stream-drop-mid-iter test: drop the stream early, verify the portal is
  closed (no leaked tx).
- Custom batch size test: `fetch_stream_with(&pool, 10)` yields rows in
  batches of 10.

**PR-3 — #[sntl::test]**
- Test the macro itself with `trybuild`: confirm expansion compiles for
  all four signatures (no args / migrations only / migrations + fixtures /
  fixtures only).
- Integration test (live-PG): run two `#[sntl::test]` fns in the same
  process, verify they get independent DBs.
- Fixture-ordering test: fixtures referenced as `fixtures("b", "a")`
  apply in that exact order (matters when `a.sql` depends on `b.sql`).
- Failure-leak test: trigger a panic in the test body, verify the test
  DB is NOT dropped and its name is printed to stderr.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| `GenericClient` trait widening breaks existing macros | All existing call sites use `&mut Connection` which still satisfies the new bound. PR-1B test suite runs the full sntl test corpus unchanged. |
| Template DB requires admin perms a user doesn't have | Document the requirement. Detect "permission denied" on first CREATE DATABASE and emit a helpful error pointing at the env var. |
| `RowStream`'s implicit tx surprises users | Doc the auto-tx behavior in `fetch_stream` and in the user guide. Test verifies drop closes the portal cleanly. |
| `&Pool` per-query checkout introduces latency for many sequential queries | Documented anti-pattern. The library doesn't prevent it — performance-conscious users hold a Connection or use `pool.begin()`. |
| `AsPool` trait conflicts with `AsRef`/other existing patterns | Trait names are deliberately distinct; review for collisions. |

## Type-naming clarification

- `sntl::Pool` is a re-export of `sentinel_driver::Pool` (existing convention,
  via `pub use sentinel_driver::Pool` in `sntl/src/lib.rs`).
- `sntl::Connection`, `sntl::PooledConnection` follow the same pattern.
- Throughout this design `sntl::Pool` and `sentinel_driver::Pool` refer
  to the same type.

## Open items resolved

1. **Test strategy:** template DB + fresh DB per test (sqlx-style, NOT
   transaction-wrap, NOT TRUNCATE).
2. **Shipping:** 3 separate PRs (PR-1 spans 2 repos, PR-2 and PR-3 each
   in sentinel only).
3. **Batch size default:** 50 for `fetch_stream`.
4. **Fixture format:** SQL files in `tests/fixtures/<name>.sql`,
   applied in declared order, single tx.
5. **Failure behavior:** test DB leaked with timestamped name on panic,
   printed to stderr for debugging.
